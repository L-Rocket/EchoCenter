## User Scenarios & Testing *(mandatory)*

### User Story 1 - Agent Navigation & Discovery (Priority: P1)

As an EchoCenter operator, I want to access a list of available AI Agents via the sidebar or dashboard so that I can quickly select which agent I want to interact with.

**Why this priority**: Core navigation requirement. Without discovery, the chat functionality is inaccessible.

**Independent Test**: Can be tested by verifying the "Agents" link in the sidebar and the existence of agent shortcuts on the dashboard.

**Acceptance Scenarios**:

1. **Given** I am on any page, **When** I click "Agents" in the sidebar, **Then** I am taken to the full list of registered agents.
2. **Given** I am on the Dashboard, **When** I see the "Active Agents" widget, **Then** I can click on a specific agent to open their interaction view.

---

### User Story 2 - Token-based Agent Authentication (Priority: P1)

As an external AI Agent developer, I want to authenticate my agent using a secure token provided by the EchoCenter Admin so that it can securely join the center and exchange messages.

**Why this priority**: Foundational security and abstraction. Treating agents as users with tokens ensures consistent auth logic.

**Independent Test**: Can be tested by attempting to connect to the WebSocket or send a message via API using a valid token and verifying attribution to the specific agent.

**Acceptance Scenarios**:

1. **Given** an agent has a valid token (generated by an Admin), **When** it connects to the system, **Then** the system identifies it as a registered "Agent User".
2. **Given** a request with an invalid or missing token, **When** an agent attempts to communicate, **Then** the system returns a 401 Unauthorized error.

---

### User Story 3 - Interactive Agent Chat (Priority: P2)

As an EchoCenter operator, I want to have a bi-directional conversation with an agent through a dedicated chat dialog using WebSockets so that I can receive real-time updates and responses.

**Why this priority**: Delivers the primary value of the feature—interaction with agents.

**Independent Test**: Open a chat dialog for "Agent A", send a message, and verify that "Agent A" receives the message and its response is displayed in the same dialog via WebSocket broadcast.

**Acceptance Scenarios**:

1. **Given** I am in an Agent's chat view, **When** I type a message and hit send, **Then** the message is displayed in the chat history.
2. **Given** an agent sends a response, **When** I have the chat dialog open, **Then** the response appears instantly in the conversation thread via WebSocket.
3. **Given** I have a conversation active, **When** I refresh the page, **Then** the chat history is cleared (Volatile Session logic).

---

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST rename all "Model" references to "Agent" in the UI and documentation.
- **FR-002**: Sidebar MUST include a primary menu item labeled "Agents".
- **FR-003**: System MUST provide an "Agent Detail" view containing a chat dialog for bi-directional communication.
- **FR-004**: Backend MUST support a new user type "AGENT" that authenticates via a long-lived API Token.
- **FR-005**: ONLY users with the `ADMIN` role MUST be able to create new Agents and generate their API Tokens.
- **FR-006**: System MUST implement a WebSocket server to handle bi-directional, real-time message exchange between users and agents.
- **FR-007**: Chat messages MUST be volatile (stored in memory/broadcasted only); the system MUST NOT persist chat history to the database.
- **FR-008**: System MUST differentiate between Human and Agent messages visually in the chat UI.

### Key Entities *(include if feature involves data)*

- **Agent**: A specialized User entity with `Role: AGENT` and a unique `AuthToken`.
- **WebSocket Message**: A real-time data packet containing `sender_id`, `receiver_id`, `payload`, and `timestamp`.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: WebSocket connection is established in under 500ms after opening the chat dialog.
- **SC-002**: 100% of messages sent via WebSocket are delivered to the recipient within 200ms (latency overhead).
- **SC-003**: Only Admins can access the "Generate Agent Token" API/UI.
- **SC-004**: Zero horizontal or vertical scroll jank in the chat dialog when receiving new messages.

## Assumptions

- **A-001**: The system will initially support 1-to-1 conversations.
- **A-002**: Agent tokens are static and manually managed by the administrator.
- **A-003**: The "Dashboard" maintains its existing 50-message persistence for status logs, while the "Chat" dialog uses volatile WebSocket messages.
