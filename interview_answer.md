# EchoCenter 面试简答（对应 interview2.md）

> 这份文档给的是“短答案”，目标是帮你先建立答题骨架。  
> 每题都尽量控制在能接住追问的长度，而不是一次答完所有细节。

---

## 一、项目总览

**1. 你先用 2 分钟介绍一下这个项目，重点讲它解决了什么问题。**  
EchoCenter 是一个集中式 AIOps 运维中枢，核心是把自然语言输入转成可执行的自动化运维流程。它解决的是传统运维里控制面分散、人工操作链路长、AI 自动化缺少安全边界这三个问题。我用 Go 做了通信和控制面，用 Butler 做调度，用 PostgreSQL 承接高并发状态持久化，再通过飞书审批把高风险动作纳入人工兜底。

**2. 为什么这个系统需要一个 centralized orchestration hub？**  
因为多节点运维场景如果没有统一中枢，会出现状态分散、权限边界模糊和执行链路难审计的问题。集中式中枢能统一做连接管理、任务路由、状态持久化和安全控制。LLM 也更适合作为中枢的决策器，而不是分散在每个节点里。

**3. 这个系统里的核心角色有哪些？Butler、普通 Agent、WebSocket Hub、数据库分别负责什么？**  
Butler 是高层调度器，负责理解用户意图并生成执行决策。普通 Agent 更像分布式执行节点，负责具体采集或操作。WebSocket Hub 负责节点长连接、消息投递和回执回传，数据库负责保存消息、任务状态、审批状态和系统配置。

**4. 你的系统本质上是一个 AI Agent 系统，还是一个被 LLM 增强的分布式控制面？**  
我更倾向于说它是一个被 LLM 增强的分布式控制面。因为真正长期存在的价值是控制面、状态流和安全边界，LLM 是提升调度和交互能力的核心组件，但不是整个系统的唯一主体。

**5. 如果让我一句话理解这个项目，你希望我怎么记住它？**  
这是一个用 LLM 做调度决策、用 WebSocket 连接分布式执行节点、并通过审批和持久化保障安全闭环的 AIOps 控制中枢。

**6. 这个项目里最难的技术挑战是什么？为什么？**  
最难的是把实时通信、AI 编排和安全控制放进同一条链路里。单独做 WebSocket、单独接 LLM、单独做审批都不难，但把三者变成一个可执行、可恢复、可审计的系统比较难。

**7. 你个人负责的部分具体到设计和实现分别有哪些？**  
我从架构到实现基本都独立完成了，包括 Go 后端、WebSocket Hub、Butler 调度链路、数据库抽象和 PostgreSQL 迁移、JWT、飞书桥接和审批流。设计上我重点做的是模块边界、状态流和安全策略；实现上我把关键主链路都打通了。

**8. 你最自豪的一项设计决策是什么？**  
我最自豪的是没有把它做成“会聊天的 AI”，而是做成“可控的自动化执行链路”。具体就是把 LLM 放在 Butler 上层决策，把实际执行、审批、状态持久化和回执做成确定性流程。

## 二、WebSocket 与高并发

**9. 你说单机支撑了 20K+ simulated concurrent connections，这个 benchmark 的测试口径是什么？**  
这个口径是空闲长连接容量，不是高频业务消息吞吐。测试重点是连接建立成功率、持有期间稳定性、RSS 内存和线程数等指标。

**10. 这些连接是空闲长连接，还是有持续业务流量的连接？**  
主要是空闲长连接，带周期性保活，不是持续高频业务消息。这个结论能证明连接层承载能力，但不能直接外推业务吞吐能力。

**11. 400MB 左右的 memory footprint 是怎么测出来的？**  
是在压测期间观测后端进程 RSS 的稳定区间得到的，而不是理论估算值。这个数字反映的是在该场景下的整体进程内存表现。

**12. 为什么你认为这个数字可信？**  
因为它是在同样测试口径下多次观测得到的稳定结果，而不是单次偶然数据。我也会明确说明它对应的是 idle connection 场景，边界比较清楚。

**13. 你在 Gorilla WebSocket 层做了哪些设计，才把内存控制在这个水平？**  
核心是比较轻量的连接对象、每连接独立 send channel、统一 Hub 管理，以及对业务消息大小和缓冲做控制。更重要的是没有在每个连接上挂太重的状态。

**14. 你的连接管理模型是什么？Hub 是怎么管理 register、unregister 和 broadcast 的？**  
我是典型的中心 Hub 模型，通过 channel 统一管理连接注册、注销和广播。这样连接生命周期、路由和异常处理会比较集中，也更容易做后续观测。

**15. 消息路由是广播、定向投递还是混合模式？为什么这么设计？**  
是混合模式。系统消息和部分监控事件适合广播，但用户与 Agent 的执行消息、审批消息更适合定向投递，这样更符合权限和性能要求。

**16. 如果某个节点消费很慢、send channel 堵住了，你怎么处理？**  
我倾向于把慢消费者视为异常连接，避免它拖垮整体 Hub。常见策略是检测 send channel 堵塞后主动摘除或断开连接，保证系统整体健康。

**17. 如果目标 Agent 不在线，Butler 下发任务时系统会怎么表现？**  
系统不应该假装执行成功，而是明确返回目标不可达。更好的做法是提前检测在线状态，若离线则拒绝下发或标记为失败，并把原因回显给用户。

**18. 这 20K 的结果为什么不能直接等价为“系统具备高吞吐能力”？**  
因为连接承载能力和业务消息吞吐能力是两回事。前者主要考验连接对象和心跳，后者会受消息处理、数据库写入、网络 I/O 和调度逻辑影响。

**19. 如果从空闲连接场景切换到高频消息场景，系统最先遇到的瓶颈会是什么？**  
大概率会先遇到网络 I/O、消息处理和持久化路径的瓶颈，而不是连接数本身。尤其是数据库写入和慢消费者问题会更早暴露。

**20. 如果让你把这个连接层继续往上推到 50K 或更高，你会优先改哪几处？**  
我会先做连接分片和水平扩展，然后优化广播和路由路径，再引入更明确的背压和流控策略。如果继续上量，单 Hub 模型通常需要拆分。

## 三、AI Coordinator、Eino 与多 Agent 编排

**21. 你说 built a core AI coordinator，它具体负责什么？**  
它负责把用户意图翻译成可执行动作，包括读取系统态、选择 Agent、决定串行还是并行、生成下一步任务，并在结果回来后继续做汇总或推进。

**22. 用户输入一句自然语言之后，到最终某个 Agent 真正执行，中间完整链路是什么？**  
先进入 Butler，会话上下文和系统态被组装进 prompt，然后模型决定下一步。如果需要调用外部能力，就通过 tool 或调度层下发到目标 Agent，回执回来后再进入 Butler 做总结或继续决策。

**23. 你为什么选 Eino，而不是直接基于 OpenAI SDK 做 function calling？**  
因为我需要的不只是调模型，而是更稳定的编排骨架。Eino 更适合组织 message、tool calling、流式输出和后续扩展，比直接手搓更容易维护。

**24. 你的系统现在更接近 ReAct 模式，还是 workflow 模式？**  
当前更接近 ReAct 驱动的调度，但从架构演进上我会把它往 workflow 化推进。简单说，现在是“模型决策主导”，后面会变成“模型给计划，执行器跑 workflow”。

**25. Butler 怎么判断当前任务应该走串行模式还是并行模式？**  
核心看动作之间是否有依赖。若上一步结果决定下一步，就走串行；若多个动作彼此独立、只是为了收集多源信息，就适合并行。

**26. 你的多 Agent workflow 是怎么表示的？是隐式的，还是有显式 plan？**  
第一阶段通常是隐式 plan，由 Butler 在上下文里驱动下一步。更成熟的方案是显式 plan，把 steps、dependencies 和 join policy 表达出来。

**27. 如果多个 Agent 都可能处理一个任务，Butler 怎么选目标节点？**  
应该基于能力标签、当前在线状态、负载、环境和最近执行表现来选，而不是单靠名字。也就是说，调度决策需要 runtime state 支撑。

**28. 你的这些 Agent 更像“智能体”，还是更像“分布式执行节点”？为什么？**  
我更把它们看作分布式执行节点。它们长期价值在于可用性、能力标签和执行结果，而不是各自独立思考。

**29. Butler 为什么不能直接暴露所有工具给模型？**  
因为 tool 太多会降低选择质量，还会吃掉上下文 token。更重要的是很多工具在当前阶段并不应该被看见。

**30. tool 太多会带来什么问题？你会怎么控制 tool 的数量和粒度？**  
会带来语义重叠、误选和 token 浪费。我的原则是对 Butler 暴露少量高层、边界清晰的 tool，把复杂性压到 tool 后端。

**31. 你怎么看 progressive tool disclosure？在你的场景里有必要吗？**  
我认为很有必要。因为诊断阶段、执行阶段、验收阶段需要的工具集合不同，按任务阶段和证据充分度渐进式暴露更稳。

**32. 如果要支持并行 workflow，Butler 应该输出什么样的结构，而不是只输出一个 next tool call？**  
应该输出结构化 plan draft，至少包含 mode、actions、depends_on 和 join policy。这样执行层才能把一轮决策转成多个并发动作。

**33. Coordinator 如果存在，它和 Butler 的职责边界应该怎么划分？**  
Butler 负责高层意图理解和下一步决策，Coordinator 负责 plan 展开、并行调度、结果汇总和上下文压缩。这样可以把语义推理和执行编排拆开。

**34. 多个 Agent 返回结果之后，谁来做结果汇总和中间态压缩？为什么？**  
更合理的是 Coordinator 或 runtime 来做第一层结构化汇总，然后 Butler 再做高层总结。这样 Butler 不需要反复面对大量原始回执。

**35. 在你的系统里，哪些部分应该由 LLM 决策，哪些部分必须由程序硬编码约束？**  
任务理解、下一步建议、结果总结适合给 LLM；权限、审批、风险等级、并行约束和状态流转必须由程序硬约束。否则系统会失控。

## 四、上下文管理、Runtime 与 MCP

**36. 多 Agent 场景下，你会怎么设计上下文管理？**  
我会分层管理，而不是只存聊天历史。至少分会话上下文、任务上下文、节点运行状态和知识上下文四层。

**37. 会话上下文、任务上下文、节点运行状态、RAG 知识，这几类上下文应该怎么分层？**  
会话上下文保连续性，任务上下文保执行状态，节点运行状态支撑调度，RAG 提供长期知识证据。这四层不要混在一起。

**38. 一次模型调用前，上下文组装顺序应该是什么？为什么规则要放前面，用户最新输入要放最后？**  
我会按规则、策略、任务状态、节点状态、知识证据、会话摘要、最近窗口、当前输入这个顺序组装。这样模型先知道边界和现状，再聚焦本轮问题。

**39. 你的 runtime 是否应该是一个独立层，而不是让 Butler 自己处理所有上下文？**  
是，我更倾向于让 runtime 独立出来。这样 Butler 可以专注于高层推理，runtime 负责上下文工程和工具路由。

**40. 这个 runtime 更适合是一个小模型、规则引擎，还是两者结合？**  
我更推荐两者结合。确定性强的判断交给规则，模糊分类和摘要压缩交给小模型。

**41. runtime 应该如何决定本轮是否需要暴露 GitHub 之类的 MCP tools？**  
它不该只看关键词，而要结合任务目标、任务阶段、权限和证据充分度。GitHub 相关工具只有在当前步骤真的相关时才暴露。

**42. 什么时候应该判定“当前不需要某类 MCP tools”？**  
当它对当前步骤没有帮助、当前仍在诊断早期、权限不允许，或者只是语义上提到了但没有真实操作需求时，就不应该暴露。

**43. MCP 在这个系统里解决的是什么问题？WebSocket 解决的又是什么问题？**  
MCP 解决能力标准化、schema 描述和 tool 调用语义问题；WebSocket 解决实时连接、心跳、任务推送和结果回传问题。

**44. 为什么既需要 WebSocket，又可能需要 MCP？**  
因为一个是 transport，一个是 capability protocol。WebSocket 让节点“连着”，MCP 让 Butler “知道怎么调它们”。

**45. 你的分布式节点是直接作为 MCP server 暴露给 Butler，还是通过平台适配成 MCP-style tools？为什么？**  
我更倾向于平台适配。这样 Butler 看到统一工具形态，底层节点协议可以保持简单，平台还能做鉴权、审批和审计。

**46. MCP tool 定义里应该放什么？哪些东西不应该暴露给 Butler？**  
应该放能力名称、输入输出 schema 和使用边界。不应该把底层传输细节、重试策略、节点内部实现全暴露给 Butler。

**47. 你的 Agent 理想形态是不是应该由 runtime 渐进式披露 tools，同时做上下文压缩？为什么？**  
是，因为这样可以降低 tool 选择难度并控制上下文膨胀。Butler 看到的应该是受控的能力集合和整理后的任务状态，而不是原始全量历史。

## 五、数据库迁移与写锁竞争

**48. SQLite 的写锁竞争在你的系统里具体是怎么产生的？**  
因为聊天消息、审批状态、集成日志等多条写路径并发叠加，而 SQLite 在服务端高并发写场景下容易出现锁竞争。这种问题在轻量原型里不明显，但在持续并发下会很突出。

**49. 哪些写路径最容易打到数据库瓶颈？**  
聊天消息持久化、授权状态更新、飞书日志和事件去重都很容易形成热点写路径。这些路径叠加时，瓶颈会很快暴露。

**50. 62.3% 的 persistence failure rate 是怎么测出来的？这个口径是什么？**  
它应该是某轮固定压测场景下，以业务写请求为口径统计出来的失败占比。关键不是这个数字本身，而是它说明锁竞争已经严重影响主链路了。

**51. continuous load of 1K concurrent requests 指的是什么负载模型？**  
指的是在固定时间窗口内持续施加千级并发请求，而不是瞬时峰值单点打满。通常这种测试更能暴露持续性争用问题。

**52. 迁移 PostgreSQL 之后为什么 failure rate 可以做到 0%？**  
因为 PostgreSQL 的并发控制模型和连接池机制更适合服务端多写路径场景。它解决了 SQLite 在写锁上的结构性短板。

**53. 除了切库之外，你还做了哪些配套优化？**  
我会同时优化 repository 抽象、连接池、事务边界和关键索引。换库只是基础，稳定性提升通常来自整条写路径的收敛。

**54. 你怎么保证从 SQLite 迁移到 PostgreSQL 时，业务层基本无感？**  
核心是把驱动差异收敛到 repository 层，包括占位符、插入后返回 ID、事务和方言差异。业务层继续调用统一接口，不直接碰底层细节。

**55. SQL placeholder、自增主键、事务语义、时间类型这些差异你怎么处理？**  
我会在 repository 层做 rebind、统一主键返回接口、封装事务边界，并尽量用统一的时间处理方式。这样差异只留在存储层。

**56. 为什么你说这是“resolved severe write-lock contention”，而不是“换了数据库所以结果更好”？**  
因为问题的本质是写锁争用，而 PostgreSQL 的引入是针对这个根因做的结构性修复。不是随便换库，而是针对并发写模型不匹配做的架构调整。

**57. 如果面试官质疑“0% failure 说明不了什么，可能只是压测不够狠”，你会怎么回应？**  
我会承认它只在那个测试口径下成立，不会夸大到所有场景。但它已经足够说明迁移后稳定性显著改善，这是有价值的工程结论。

## 六、JWT、授权与飞书桥接

**58. JWT 在你的系统里主要保护哪些边界？**  
主要保护前端用户接口、管理接口和人类用户的 WebSocket 接入边界。它确保用户身份和角色信息能被统一校验。

**59. 为什么人类用户适合用 JWT，而机器节点未必直接适合？**  
因为人类用户更适合会话型身份认证，而机器节点更适合独立 token、轮换和吊销模型。两者生命周期和权限边界不同。

**60. 飞书 WebSocket bridge 在这个系统里的主要作用是什么？**  
它把企业 IM 入口和 Butler 链路打通，让消息能从飞书进入系统，同时把审批卡片和执行结果再回推到飞书。

**61. 为什么选择 Feishu WebSocket bridge，而不是纯 webhook 模式？**  
长连接模式在某些企业集成场景下更适合持续事件接入，也更方便做实时回执和状态保持。它能减少单纯回调模式下的一些集成复杂度。

**62. 你简历里说了 policy routing，这个 routing 是按什么规则做的？**  
主要按消息来源、用户身份、会话类型、命令前缀、白名单和路由策略做判断。目标是让不同类型消息进入正确处理链路。

**63. 一条来自飞书的消息，怎么安全地进入 Butler 的处理链路？**  
先验证来源和连接器配置，再做用户映射、策略过滤和去重，最后才路由到 Butler。也就是先做安全和准入，再做语义处理。

**64. 你的 enterprise-integrated authorization workflow 是怎么工作的？**  
Butler 识别到高风险动作后先写 pending approval，再通过飞书发审批卡片。审批通过后才进入执行，执行结果和状态再回写系统。

**65. 为什么高风险动作不能只依赖 RBAC，而必须要有人类审批？**  
因为 RBAC 只能回答“有没有权限”，不能回答“这次动作在当前上下文是否应该发生”。AI 自动化里最危险的是上下文不完整时的错误执行，所以必须有人类兜底。

**66. 你怎么保证审批流是幂等的？**  
关键是审批记录要有唯一 action ID，并在状态机上保证重复点击不会重复执行。审批只是状态变化，不应该无限触发动作。

**67. 如果飞书里用户批准了，但目标 Agent 此时已经离线，系统怎么保证状态一致性？**  
审批通过只代表获得执行许可，不代表动作已经成功。真正执行时再检查目标可达性，如果失败就显式写成 failed 或 not executed，并通知用户。

## 七、RAG 与知识设计

**68. 在你这个 AIOps 场景里，RAG 最合理存哪些内容？**  
最合理的是 SOP、故障复盘、历史处理经验和特殊环境说明。这些属于组织内部长期知识，对诊断和建议最有帮助。

**69. 为什么 SOP、故障复盘、历史经验适合放进 RAG？**  
因为它们是模型参数里不会天然知道的组织内部知识，而且语义检索价值高。相比实时数据，它们更稳定，也更适合作为证据注入。

**70. 为什么实时节点状态、实时告警值不适合放进 RAG？**  
因为这些是运行时真相，不是长期知识。它们应该通过实时状态源直接查询，而不是靠语义检索。

**71. 如果知识库内容过期、互相冲突或者本身质量不高，会带来什么问题？**  
会让模型“有依据地答错”，甚至比纯幻觉更危险。因为用户会误以为它引用了知识就一定可靠。

**72. 你会怎么治理这类知识库？**  
我会做版本管理、时间戳、来源标记和低质量文档清洗。高风险场景下还要强调多源一致或人工确认。

**73. 你会怎么做 context compression，让 Butler 不被长对话拖垮？**  
保留任务状态和最近窗口，对旧对话做摘要，把无关内容删掉。重点不是保留所有原话，而是保留当前任务所需的最小充分信息。

## 八、系统设计追问

**74. 如果 Butler 是单点，系统的高可用怎么做？**  
关键是让会话和任务状态外置化，然后把 Butler 做成多实例。这样故障恢复时可以续跑或至少不丢核心状态。

**75. 如果 pending action 只在内存里，会有什么风险？**  
一旦进程重启、多实例部署或审批延迟到其他实例，就会出现状态丢失和一致性问题。这是典型技术债。

**76. 如果系统规模从几十个 Agent 增长到几千个，你最先重构哪一层？**  
我会优先重构连接与调度层，包括连接分片、节点状态管理和异步消息路径。因为这会最先成为系统上限。

**77. 如果现在要支持更复杂的并行与串行组合 workflow，你会引入独立 workflow engine 吗？**  
会考虑，因为复杂度继续上升后，单纯靠 ReAct 隐式驱动不够稳。独立 workflow engine 更适合表达 DAG、重试和补偿。

**78. 你的系统里最大的技术债是什么？**  
最大的技术债通常是内存态上下文和单点调度耦合。它在单机阶段可接受，但会限制恢复能力和横向扩展。

**79. 哪些模块现在更像“可行性方案”，哪些已经接近“生产级设计”？**  
连接层、JWT 和基本持久化比较接近生产级；而多 Agent 的复杂 workflow、runtime 分层和高可用更多还是阶段性方案。承认这个边界会更专业。

**80. 如果再给你两个月，你会优先补哪三件事？**  
我会优先补 runtime/context 分层、pending action 持久化和 workflow engine 化。因为这三件事最直接决定系统能不能往生产级走。

## 九、行为与 Ownership

**81. 这个项目里你做过的一个错误决策是什么？后来怎么修正的？**  
我会说早期可能把太多状态压在 Butler 内部或单机内存里，导致系统边界不够清晰。后来逐步把存储、审批和调度状态收敛成更明确的结构。

**82. 你在哪个点上体现了 end-to-end ownership？**  
我不是只做了某个模块，而是把从连接、调度、存储、安全到企业集成这条主链路都推进到了可运行状态。遇到瓶颈时也不是绕开，而是主动切数据库和补安全链路。

**83. 如果这个项目线上出问题，你最先会看哪些指标和日志？**  
我会先看连接数、消息投递失败、Hub 堵塞、数据库错误、Agent 回执超时和审批状态异常。这些指标最能反映主链路是否卡住。

**84. 这个项目最能证明你具备什么能力？**  
最能证明的是端到端后端系统设计和落地能力，尤其是把 AI 编排放进一个可执行、可控、可审计的系统里，而不是停留在 demo 层。

**85. 如果我是微软中国的面试官，我为什么应该相信你能独立扛一个复杂后端系统？**  
因为这个项目本身就要求我同时处理通信、并发、状态一致性、数据库迁移、安全和企业集成，而且我不是只把功能拼出来，而是不断识别和修正系统性问题。

## 十、高概率连续追问

**86. 这个结果的边界是什么？你能证明什么，不能证明什么？**  
我会主动讲清楚 benchmark 口径，比如空闲长连接不等价于高吞吐，0% failure 只在特定压测场景下成立。能证明的是在该场景下的稳定性改善，不能证明所有生产场景都如此。

**87. 哪一步是 LLM 决定的，哪一步是程序强约束的？**  
LLM 决定任务理解、下一步建议和结果总结；程序强约束身份、权限、审批、路由、状态机和执行边界。这条分界线必须清楚。

**88. 你的系统最先会在哪个地方失效？**  
如果流量和复杂度继续上涨，最先暴露的问题通常是单点调度和内存态上下文，而不是模型本身。其次是消息路径和数据库热点写入。

**89. 你这个项目为什么不是一个“套了 AI 的 Demo”？**  
因为我打通的是一条完整可执行链路，而不是只做聊天界面。系统里有连接层、调度层、状态持久化、审批链路和企业集成，这些才是从 demo 跨到系统的关键。

**90. 如果今天把这个系统交给一个新团队，你觉得他们最容易踩什么坑？**  
最容易踩的是把所有逻辑重新堆回 Butler，导致上下文、执行和状态边界再次混乱。第二个坑是低估并发写入和状态一致性问题，第三个是把 tool 暴露做得太多太杂。
